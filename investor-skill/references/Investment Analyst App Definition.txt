AI-Powered Financial Advisor System
1. Executive Summary
This document defines the full scope, purpose, and design philosophy of an AI‑powered Financial Advisor System. The project aims to deliver a personalized, constraint‑aware, learning financial advisory platform that produces daily investment insights and recommendations based on each user’s unique preferences, ethical constraints, goals, and portfolio structure.
Unlike traditional robo‑advisors that rely on static rules or generic optimization, this system is designed as a dynamic, adaptive advisor. It observes its own historical recommendations, compares them with hypothetical and real portfolio outcomes, and continuously improves decision quality over time.
The platform supports both scheduled advisory outputs (daily reports) and interactive inference mode, allowing users to query hypothetical scenarios and portfolio alternatives.
________________


2. Project Purpose and Vision
2.1 Purpose
The core purpose of the project is to:
* Assist individuals in making better, more informed financial decisions
* Respect user‑defined constraints such as Shariah compliance or ethical investment rules
* Provide clear, calm guidance rather than constant trading pressure
* Learn from past recommendations to improve future advice
The system is advisory in nature. It does not execute trades automatically and does not enforce actions. The final decision always remains with the user.
2.2 Vision
The long‑term vision is to create a trustworthy digital financial companion that:
* Understands the user’s values as well as their numbers
* Explains market conditions in human‑readable language
* Evolves its strategy over time based on historical outcomes
* Can simulate alternative financial realities (“what if” scenarios)

4. Core System Capabilities
4.1 Personalized User Profiling
Each user defines a financial profile that includes:
* Investment goals (growth, preservation, income, balance)
* Time horizon (short / medium / long term)
* Risk tolerance
* Ethical or regulatory constraints (e.g. Shariah compliance)
* Asset preferences and exclusions
* Current portfolio allocation
This profile becomes a persistent decision context for all advisory outputs.
________________


4.2 Regular Advisory Reports (Every 3 Days)
The system produces advisory reports every 3 days that may include:
* Market environment summary
* Asset‑level observations (Gold & Silver, Global Stocks & Funds, Turkish Stocks & Funds)
* Portfolio impact analysis
* Risk commentary
* Recommendation section

Recommendation Types
The advisor is not forced to act every cycle. Possible outputs include:
* Buy / Increase exposure
* Reduce / Exit exposure
* Hold
* Wait / No action recommended
When no action is advised, the system provides a rationale and environmental overview to maintain user confidence and understanding.

4.2.1 Data & Tracking Schedule
* Stock & Fund Pool Initialization: Run once every 2 weeks to ensure compliance (S&P 500 Shariah, BIST 100 Katılım, MSCI ACWI Islamic Index, No Israel connection).
* Price & Financial Data Updates: Run once daily using yfinance to gather rich fundamental datasets (ebitda_ev, ep_ttm, bp_ttm, cfp_ttm, roe, roe_stability, revenue_ttm_yoy_growth, net_profit_ttm_yoy_growth, residual_volatility, liquidity_turnover_weighted).
* Main Generative Pipeline (Research -> Discussion -> Decision): Run once every 3 days.
________________


4.3 Portfolio‑Aware Decision Making
The advisor always reasons relative to the user’s actual portfolio, not in isolation.
For each recommendation:
* Current portfolio weights are evaluated
* Proposed portfolio weights are calculated
* Risk and constraint compliance is verified
No recommendation is made without validating that it fits within the defined constraints.

5. Historical Tracking and Learning Mechanism
5.1 Recommendation History Storage
For every advisory cycle, the system stores:
* Date and market context
* User portfolio state at the time
* Advisor’s recommended actions
* Hypothetical post‑recommendation portfolio
* Asset prices at decision time
This creates a complete decision audit trail.
________________


5.2 Base Portfolio vs Advisor Portfolio Tracking
Two parallel portfolio timelines are maintained:
1. Base Portfolio
The user’s actual allocation at the time of advice
2. Advisor‑Suggested Portfolio
A hypothetical portfolio assuming the advisor’s recommendation was followed
These timelines allow:
   * Performance comparison over time
   * Measurement of advisory quality
   * Strategy refinement and tuning
________________


5.3 Adaptive Tuning
Historical outcomes are used to:
   * Detect consistently strong or weak recommendation patterns
   * Adjust confidence levels and weighting of signals
   * Improve future decision logic
This learning process is incremental and explainable, not opaque or purely statistical.

6. Inference and Interactive Query Mode
Beyond daily reports, the system includes an inference mode where users can ask direct questions.
6.1 Example Queries
   * “What is my portfolio’s performance over the last 6 months?”
   * “If I had invested all my capital into asset X, what would my current value be?”
   * “How much risk am I exposed to right now?”
   * “What happens if I rebalance toward safer assets?”
________________


6.2 Imaginary Asset Tracking
The system supports hypothetical portfolio simulations, including:
   * Single‑asset full allocation scenarios
   * Alternative allocation strategies
   * Counterfactual historical performance
These simulations do not affect real portfolio history but are stored separately for reasoning and explanation.

   * Asset‑level return calculation

   * Period‑based comparisons

7.2 Performance Analytics

8. Transparency and Explainability
A core design principle is explainability.
Each advisory output includes:
   * Why a recommendation was made (or not made)
   * Which constraints influenced the decision
   * What risks are being considered
   * What uncertainties remain
The system avoids “black box” behavior and favors reasoned narratives.

9. Non‑Goals and Boundaries
The system explicitly does not:
   * Execute trades automatically
   * Guarantee returns
   * Replace licensed financial advisors
   * Provide legal or tax advice
It is designed as a decision support system, not an autonomous trader.

10. Learning, Adaptation, and Advisor Self-Improvement (Future Work)
10.1 Motivation
A key long-term goal of the project is to enable the financial advisor to improve the quality of its recommendations over time by learning from past advisory actions and their outcomes.
The advisor is not intended to be static or rule-based indefinitely. Instead, it should gradually evolve into a self-reflective system that understands which types of guidance were beneficial, neutral, or harmful under different market regimes and portfolio contexts.
This learning capability is considered strategically important but methodologically open, and is therefore classified as future work / desired functionality.


10.2 Learning Scope and Boundaries
The learning mechanism is explicitly constrained by the following principles:
   * Learning focuses on advisory quality, not raw profit maximization

   * The system remains human-interpretable and auditable

   * No autonomous execution of trades

   * User-defined constraints (e.g. Shariah compliance, exclusions) are never overridden

   * Short-term noise must not dominate long-term signal

The advisor learns how to advise better, not how to trade aggressively.


10.3 Feedback and Evaluation Signals
The system already stores rich historical data that can be reused as learning signals, including:
      * Difference between:

         * Base portfolio performance

         * Advisor-suggested hypothetical portfolio performance

            * Asset-level return contribution after recommendations

            * Drawdowns and volatility following advisory actions

            * Time-delayed outcomes (e.g. 7 / 30 / 90 days after advice)

            * Frequency of “hold / wait” recommendations vs active suggestions

            * User overrides or ignored recommendations (if available)

These signals form the foundation of any learning or reward mechanism.


10.4 Candidate Learning Approaches (Exploratory)
At this stage, multiple learning strategies are intentionally kept open.
A. Prompt-Level Adaptive Learning
A lightweight and early-phase approach where:
               * Past advisory outcomes are summarized structurally

               * These summaries are injected into agent prompts

               * Prompt emphasis is adjusted (risk sensitivity, conviction thresholds, caution)
Pros
                  * Low complexity

                  * Fully interpretable

                  * Compatible with API-based reasoning models
Cons
                     * Indirect learning

                     * No explicit optimization objective

B. Reinforcement Learning via Verbal or Scalar Rewards (RLVR)
The advisory loop is reframed as a sequential decision process:
                        * State: portfolio + market context + constraints

                        * Action: advisory recommendation

                        * Reward: relative portfolio performance, risk-adjusted metrics, regret signals

Rewards may be:
                           * Numeric (performance deltas, drawdown penalties)

                           * Verbal/structured (good / neutral / poor outcome labels)

Pros
                              * Clear feedback loop

                              * Objective-driven improvement

Cons
                                 * Reward design is non-trivial

                                 * Risk of overfitting to historical regimes

C. Multi-Agent Reinforcement Learning (MARL) Extension
Given the existing architecture, learning can be extended to the multi-agent level:
                                    * Research agents, discussion agents, and decider agent receive partial credit

                                    * Coordination quality becomes a learning objective

                                    * Signal weighting across agents evolves over time

This reframes the system as a cooperative MARL advisory ecosystem.
10.5 Roadmap Positioning
Learning and self-improvement are categorized as:
                                       * Desired but non-blocking

                                       * Research-driven

                                       * Introduced incrementally

A realistic progression is:
                                          1. Static advisor with full decision logging

                                          2. Prompt-level adaptive feedback

                                          3. Offline evaluation and reward modeling

                                          4. Controlled RLVR experiments

                                          5. Optional MARL-based optimization

12. Conclusion
This project defines a modern, adaptive, constraint‑aware financial advisor that prioritizes clarity, personalization, and learning over time. By combining daily advisory outputs, historical self‑evaluation, and interactive inference, the system aims to become a long‑term financial companion rather than a short‑term trading tool.
This document serves as the foundational specification for design, development, and iteration of the platform.
