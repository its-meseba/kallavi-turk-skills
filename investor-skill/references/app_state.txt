0. V2 ARCHITECTURE UPDATE (FEB 2026)
================================================================================

This document was originally written for the V1 architecture. The application
has since been upgraded to V2.

Completed Tasks in V2:
  * REMOVED Real Estate Domain: The real estate asset class was completely
    removed from the research and definitions because it lacked daily liquidity
    and accurate daily API pricing data.
  * SPLIT Stocks & Funds: The monolithic "Stocks & Funds" category was split
    into two distinct research silos: "Global Stocks & Funds" and
    "Turkish Stocks & Funds".
  * INTEGRATED YFINANCE METRICS POOL: A new script (`update_pool.py`) was
    created to fetch real-time fundamental metrics (P/E, ROE, EBITDA) via
    the yfinance API to supply quantitative grounding to the agents.
  * MIXTURE OF EXPERTS (MoE) DISCUSSION: The discussion phase was rewritten.
    Instead of 3 fixed agents debating iteratively, the system now spawns 10
    dynamic Discussion Agents in a single pass. Each agent is randomly assigned
    an "Aggression Level" and "Risk Appetite".
  * LONG-TERM DECIDER MEMORY: The Decider agent now receives the 3 most
    recent final decision reports, preventing strategic flip-flopping.
  * PORTFOLIO PRIVACY: The `portfolio/` directory was added to `.gitignore`.

Pending Tasks / Future Directions:
  * MARL and RLHF: Add automated logic to backtest Decider agent recommendations
    and supply it as learning parameters on future runs.
  * Add automatic TEFAS scraping to `update_pool.py` for Turkish mutual funds.

1. EXECUTIVE SUMMARY
================================================================================

The Investment Advisor Multi-Agent System is a Python-based, AI-powered
financial advisory platform that produces periodic investment guidance for a
single user. It employs a pipeline of 14 specialized AI agents (powered by
Google Gemini models) organized into three phases — Research, Discussion, and
Decision — culminating in an emailed advisory report.

The system is constraint-aware (Halal/Shariah compliance, No-Israel exposure),
currency-aware (TRY/USD/EUR), and portfolio-aware (dynamically injecting the
user's current holdings into agent prompts). It tracks a "shadow" advisory
portfolio alongside the user's actual portfolio to measure advisory quality
over time, and supports an interactive inference mode for ad-hoc questions.

The current codebase is functional, well-structured, and already generating
live advisory reports. However, it operates as a sequential, script-based
pipeline with no persistent agent memory, no true inter-agent communication
protocol, and limited learning capability. This report details the current
state, identifies gaps, and proposes concrete tools, frameworks, and methods
to evolve the system into a more sophisticated, adaptive advisory platform.

2. PROJECT VISION AND GOALS
================================================================================

As defined in the foundational "Investment Analyst App Definition" document,
the project aims to create a:

  * PERSONALIZED financial advisor that respects user constraints (Shariah
    compliance, ethical exclusions, risk tolerance, time horizon)
  * DAILY advisory engine producing structured, actionable investment guidance
  * PORTFOLIO-AWARE decision-maker that reasons relative to the user's actual
    holdings, not in isolation
  * SELF-IMPROVING system that tracks its own recommendations against outcomes
    and learns from past performance
  * INTERACTIVE companion that supports "what-if" scenarios and ad-hoc queries
  * TRANSPARENT and EXPLAINABLE advisor — no black-box behavior

The system explicitly does NOT:
  * Execute trades automatically
  * Guarantee returns
  * Replace licensed financial advisors
  * Provide legal or tax advice

The long-term vision is a self-reflective, multi-agent cooperative system that
gradually improves its advisory quality through structured feedback loops.

3. CURRENT SYSTEM ARCHITECTURE
================================================================================

--------------------------------------------------------------------------------
3.1 High-Level Pipeline
--------------------------------------------------------------------------------

The system follows a linear, phased pipeline executed via main.py:

  Phase 1: RESEARCH
    9 specialized agents gather real-time market data via Google Search
    grounding. Three asset classes (Real Estate, Gold/Silver, Stocks/Funds)
    are each covered by three sub-specialists (News, Fundamentals, Sentiment).

  Phase 2: DISCUSSION
    3 discussion agents (each biased toward one asset class) synthesize the
    9 research reports. They iterate N times (default: 3), with each iteration
    receiving the previous round's outputs from the other two agents, enabling
    a form of structured debate and perspective refinement.

  Phase 3: DECISION
    1 decider agent ingests all 12 inputs (9 research + 3 discussion) plus
    the last 3 historical decision reports. It self-iterates M times (default:
    3) through phases of initial synthesis, critical review, and final
    refinement to produce the final advisory report.

  Phase 3.5: ADVISORY EXECUTOR
    An LLM-assisted parser extracts structured BUY/SELL/HOLD actions from the
    decider's report and executes them on a "shadow" (advisory) portfolio,
    enabling performance tracking of the advisor's recommendations.

  Phase 4: EMAIL DELIVERY
    The final decision report is converted to styled HTML and sent via Gmail
    SMTP to the configured recipient.

  Separately: INFERENCE MODE (inference.py)
    An interactive Q&A agent that uses the last N final reports as context
    plus real-time web search to answer ad-hoc investment questions.

  Separately: PRICE FETCHER + PORTFOLIO UPDATER
    Scripts that fetch live prices from yfinance/TEFAS and update both the
    current and advisory portfolios with latest market values.

  Separately: EVALUATION
    Logs daily portfolio values to CSV and generates comparison plots
    (current vs. advisory portfolio, asset prices, portfolio composition).

--------------------------------------------------------------------------------
3.3 Data Flow and State Management
--------------------------------------------------------------------------------

  * In full pipeline runs, data flows IN-MEMORY between phases
  * Reports are also persisted to disk with date-stamped filenames
  * For partial runs (e.g., --decider-only), data is loaded from disk using
    today's date as a filter
  * Portfolio state is stored in JSON (portfolio/current_portfolio.json)
  * Price history is stored in TSV format (tracking/price_history.txt)
  * Evaluation metrics are stored in CSV (tracking/evaluation_log.csv)
  * Advisory actions are logged in JSON (tracking/advisory_actions_log.json)

There is NO database, NO message queue, and NO persistent agent memory beyond
file-based storage.

4. DETAILED IMPLEMENTATION ANALYSIS
================================================================================

--------------------------------------------------------------------------------
4.1 Research Phase (9 Agents)
--------------------------------------------------------------------------------

Each research agent:
  * Loads a dedicated prompt file defining its role, asset class, and sub-focus
    (News/Fundamentals/Sentiment)
  * Uses Google Search grounding to access real-time web data
  * Generates a structured report following a template defined in its prompt
  * Implements retry logic with exponential backoff for rate limiting (429 errors)
  * Saves reports with date-stamped filenames to reports/research/

Agents run SEQUENTIALLY with a configurable delay (default: 10 seconds) between
agents to avoid API rate limits. This means the research phase alone takes
approximately 2-3 minutes minimum.

Key Observation: Research agents do NOT share context with each other. Each
operates independently, which can lead to redundant searches and missed
cross-asset correlations.

--------------------------------------------------------------------------------
4.2 Discussion Phase (3 Agents, N Iterations)
--------------------------------------------------------------------------------

Each discussion agent:
  * Has a "biased perspective" (Gold-favored, Real Estate-favored, Stocks-favored)
  * Receives ALL 9 research reports as input
  * In iteration > 1, also receives the OTHER two agents' previous outputs
    (not its own, to avoid self-reinforcement)
  * Portfolio data is dynamically injected into the system prompt
  * Runs for N iterations (default: 3), creating a "structured debate"

This design is a form of "Society of Mind" multi-agent deliberation. Each agent
argues for its preferred asset class but must engage with the research data and
the other agents' perspectives.

Key Observation: Agents don't truly "communicate" — they receive batched text
outputs from previous rounds. There is no negotiation protocol, no shared
workspace, and no explicit agreement/disagreement tracking.

--------------------------------------------------------------------------------
4.3 Decider Phase (1 Agent, Self-Iterating)
--------------------------------------------------------------------------------

The decider agent:
  * Receives all 12 inputs (9 research + 3 final discussion outputs)
  * Also receives the last 3 historical decision reports for consistency
  * Self-iterates through 3 phases:
    - Iteration 1: Initial Synthesis
    - Iteration 2: Critical Review (challenge own assumptions)
    - Iteration 3: Final Refinement
  * Produces a structured report with: Market Overview, Key Suggestions,
    Market Expectations, Candidate New Investments, Risk Alerts
  * Has explicit rules for long-term consistency (no flip-flopping)
  * Output is designed for email delivery (bullet points, no tables)

Key Observation: The decider is the most sophisticated agent, but its
"self-iteration" is stateless — each iteration is a fresh API call with the
previous output pasted in. There is no persistent working memory or scratchpad.

--------------------------------------------------------------------------------
4.4 Advisory Executor (Shadow Portfolio)
--------------------------------------------------------------------------------

The advisory executor:
  * Uses an LLM (gemini-2.0-flash) to parse the decider's free-text report
    into structured JSON actions (BUY/SELL/HOLD)
  * Executes these actions on the advisory (shadow) portfolio using Python math
  * Handles compound actions (e.g., REALLOCATE = SELL + BUY)
  * Implements fuzzy asset name matching
  * Logs all executed and skipped actions
  * Manages a cash pool for proceeds from sales

This is a critical component for the learning loop — it creates the "what if
we followed the advisor" portfolio that can be compared against the user's
actual portfolio.

Key Observation: The LLM parsing step introduces fragility. JSON extraction
from free text can fail despite retry logic. A more structured output format
from the decider (e.g., machine-readable JSON alongside human-readable text)
would be more robust.

--------------------------------------------------------------------------------
4.5 Inference Mode (Interactive Q&A)
--------------------------------------------------------------------------------

The inference agent:
  * Loads the last N final reports as context (default: 3)
  * Has Google Search grounding for real-time data
  * Runs in an interactive loop (REPL-style) or single-question mode
  * Uses gemini-2.5-pro for deep reasoning

Key Observation: The inference agent is NOT portfolio-aware (unlike discussion
and decider agents). It doesn't receive the user's current portfolio, which
limits the personalization of responses.

--------------------------------------------------------------------------------
4.6 Price Fetching and Portfolio Updates
--------------------------------------------------------------------------------

  price_fetcher.py:
    * Fetches prices from yfinance (global stocks, forex, commodities)
    * Fetches TEFAS fund prices for Turkish mutual funds
    * Converts all prices to USD using live USD/TRY rate
    * Appends to price_history.txt (idempotent — overwrites today's row)
    * Also saves latest_prices.json for other scripts

  update_portfolio.py:
    * Reads latest prices from price_history.txt
    * Updates both current and advisory portfolios
    * Recalculates all derived fields (totals, percentages)
    * Does NOT change piece counts (only prices)

Key Observation: Real estate prices are NOT fetched automatically. A cached
default value ($150,000 USD for Cankaya average) is used. This is a known
limitation acknowledged in the code.

--------------------------------------------------------------------------------
4.7 Evaluation and Visualization
--------------------------------------------------------------------------------

  evaluate.py:
    * Logs daily portfolio values to evaluation_log.csv
    * Generates 3 matplotlib charts comparing current vs. advisory portfolios
    * Uses seaborn-style plotting with auto log-scale for wide price ranges

Key Observation: The evaluation is purely visual and numeric. There is no
automated analysis of advisory quality, no statistical significance testing,
and no feedback from evaluation results back into the advisory pipeline.

5. STRENGTHS OF THE CURRENT IMPLEMENTATION
================================================================================

  1. WELL-STRUCTURED MULTI-AGENT ARCHITECTURE
     Clear separation of concerns: research, discussion, decision. Each agent
     has a defined role, dedicated prompt, and configurable parameters.

  2. ITERATIVE REFINEMENT
     Both the discussion phase (inter-agent iteration) and the decider phase
     (self-iteration) employ multiple rounds of refinement, improving output
     quality beyond single-shot generation.

  3. HISTORICAL AWARENESS
     The decider receives its last 3 reports, enabling consistency and
     preventing contradictory advice across cycles.

  4. DUAL PORTFOLIO TRACKING
     The shadow advisory portfolio is a powerful mechanism for measuring
     advisor quality — a feature most personal advisory systems lack entirely.

  5. CONSTRAINT-AWARE DESIGN
     Halal/Shariah compliance and No-Israel exposure are first-class concerns
     woven throughout the prompt system.

  6. CLEAN CODE AND CONFIGURATION
     Centralized config.py, consistent coding patterns, proper error handling
     with retry logic, and clear documentation.

  7. REAL-TIME DATA ACCESS
     Google Search grounding provides research agents with live market data,
     while yfinance/TEFAS provide precise price feeds.

  8. FLEXIBLE EXECUTION MODES
     Full pipeline, phase-by-phase, and interactive inference modes provide
     versatile usage options.


================================================================================
6. GAPS, LIMITATIONS, AND WEAKNESSES
================================================================================

  1. NO TRUE INTER-AGENT COMMUNICATION
     Agents don't negotiate, request clarification, or share structured
     signals. They receive batched text and generate text — no protocol.

  2. NO PERSISTENT AGENT MEMORY
     Each agent invocation is stateless. There is no vector store, no
     episodic memory, and no accumulated world model.

  3. SEQUENTIAL EXECUTION
     All agents run sequentially. The 9 research agents could run in parallel,
     potentially reducing pipeline time by 70-80%.

  4. SINGLE LLM PROVIDER DEPENDENCY
     The entire system depends on Google Gemini. An outage or API change
     would halt all operations.

  5. NO LEARNING FEEDBACK LOOP
     Evaluation data (current vs. advisory portfolio performance) is not fed
     back into the advisory pipeline. The system generates advice but never
     learns from whether that advice was good or bad.

  6. FRAGILE ACTION PARSING
     The advisory executor relies on LLM-based free-text-to-JSON conversion,
     which is inherently unreliable even with retry logic.

  7. NO USER INTERFACE
     All interaction is CLI-based. There is no web dashboard, no mobile
     interface, and no visualization of reports beyond email.

  8. LIMITED FINANCIAL ANALYTICS
     No Sharpe ratio, no maximum drawdown calculation, no correlation matrix,
     no Monte Carlo simulation. The evaluation is limited to raw value
     comparison.

  9. NO REAL ESTATE PRICE AUTOMATION
     Real estate prices are hard-coded defaults, unlike all other asset
     classes which are fetched automatically.

  10. NO RATE LIMIT MANAGEMENT ACROSS PIPELINE
      Rate limiting is handled per-agent with hardcoded delays. A centralized
      token bucket or quota manager would be more efficient.

  11. NO TESTING
      There are no unit tests, integration tests, or automated prompt
      evaluation harnesses.

  12. INFERENCE AGENT NOT PORTFOLIO-AWARE
      Unlike discussion and decider agents, the inference agent doesn't
      receive the user's current portfolio context.


================================================================================
7. FUTURE DIRECTIONS
================================================================================

--------------------------------------------------------------------------------
7.1 Short-Term Improvements (0-3 Months)
--------------------------------------------------------------------------------

  A. PARALLEL RESEARCH EXECUTION
     Use Python asyncio or ThreadPoolExecutor to run research agents in
     parallel, reducing pipeline time from ~10+ minutes to ~2-3 minutes.

  B. STRUCTURED DECIDER OUTPUT
     Have the decider produce a machine-readable JSON block alongside its
     human-readable report, eliminating the need for LLM-based parsing in
     the advisory executor.

  C. INJECT PORTFOLIO INTO INFERENCE AGENT
     Add portfolio context to the inference agent's prompt for personalized
     responses to ad-hoc questions.

  D. BASIC FINANCIAL METRICS
     Add Sharpe ratio, maximum drawdown, annualized return, and volatility
     calculations to the evaluation module.

  E. ADD UNIT TESTS AND PROMPT EVALUATION
     Create test harnesses for agent outputs, portfolio calculations, and
     file handling. Implement LLM-as-judge evaluation for report quality.

  F. CENTRALIZED LOGGING
     Replace scattered print statements with Python's logging module
     configured with file and console handlers.

--------------------------------------------------------------------------------
7.2 Medium-Term Enhancements (3-9 Months)
--------------------------------------------------------------------------------

  A. PROMPT-LEVEL ADAPTIVE LEARNING (Phase 1 of Learning Roadmap)
     Feed summarized evaluation data (advisory vs. actual performance) into
     agent prompts. Example: "In the last 30 days, your SELL recommendations
     outperformed your BUY recommendations by 12%. Consider this pattern."

  B. VECTOR MEMORY / RAG SYSTEM
     Implement a vector database (ChromaDB, Pinecone, or FAISS) to store
     and retrieve historical reports, market events, and advisory outcomes.
     Agents can then query relevant historical context rather than receiving
     only the last 3 reports.

  C. WEB DASHBOARD
     Build a Streamlit or Gradio web interface for:
     - Portfolio visualization with interactive charts
     - Report browsing and comparison
     - Inference mode with chat UI
     - Advisory performance metrics dashboard

  D. MULTI-MODEL SUPPORT
     Abstract the LLM layer to support multiple providers (OpenAI GPT-4o,
     Anthropic Claude, local models via Ollama). This provides redundancy
     and allows cost optimization.

  E. STRUCTURED AGENT COMMUNICATION PROTOCOL
     Replace raw text passing between agents with a structured message format:
     {confidence_score, key_signals, recommended_actions, risk_flags, ...}
     This enables quantitative analysis of agent agreement and disagreement.

  F. REAL ESTATE DATA AUTOMATION
     Integrate a real estate data API (or web scraping pipeline) for Turkish
     property market prices, removing the hard-coded default.

  G. NOTIFICATION SYSTEM UPGRADE
     Add Telegram bot, WhatsApp, or push notification support alongside email.

--------------------------------------------------------------------------------
7.3 Long-Term Vision (9-18+ Months)
--------------------------------------------------------------------------------

  A. REINFORCEMENT LEARNING VIA VERBAL REWARDS (RLVR)
     Frame the advisory loop as a sequential decision process:
     State = portfolio + market context + constraints
     Action = advisory recommendation
     Reward = risk-adjusted portfolio performance delta
     Use this to fine-tune the advisory prompts or model weights.

  B. MULTI-AGENT REINFORCEMENT LEARNING (MARL)
     Extend learning to the multi-agent level: assign partial credit to
     research, discussion, and decider agents. Optimize coordination quality
     and signal weighting across agents.

  C. "WHAT-IF" SIMULATION ENGINE
     A full counterfactual simulation system that can replay history with
     alternative allocation strategies, answering questions like "What if I
     had invested all capital in gold 6 months ago?"

  D. MULTI-USER SUPPORT
     Extend the system to support multiple user profiles with different
     constraints, portfolios, and risk tolerances. Enable anonymized
     benchmarking across users.

  E. REGULATORY AND COMPLIANCE MODULE
     Automated Shariah compliance screening using standardized financial
     ratios, Israel-connection databases, and ESG scoring APIs.

--------------------------------------------------------------------------------
8.1 Multi-Agent Communication Frameworks
--------------------------------------------------------------------------------

  CREWAI (crewai.com)
    * Purpose-built for multi-agent collaboration with defined roles, goals,
      and task delegation
    * Supports sequential, hierarchical, and consensual task execution
    * Built-in memory, tool use, and inter-agent delegation
    * Why it fits: Natural replacement for the current manual agent orchestration.
      Agents could delegate sub-tasks, request clarification from each other,
      and maintain shared context.
    * Consideration: Primarily OpenAI-focused, but supports custom LLM backends.

  AUTOGEN (Microsoft, github.com/microsoft/autogen)
    * Multi-agent conversation framework with flexible topologies
    * Supports group chat, two-agent chat, and nested conversations
    * Built-in code execution and human-in-the-loop capabilities
    * Why it fits: The discussion phase could be modeled as a group chat where
      agents naturally debate and converge, rather than receiving batched outputs.
    * Consideration: Strong Python-native support; Azure OpenAI integration.

  LANGGRAPH (LangChain, github.com/langchain-ai/langgraph)
    * Graph-based agent orchestration with stateful, cyclical workflows
    * Supports conditional branching, parallel execution, and human-in-the-loop
    * Persistent state across graph nodes with checkpointing
    * Why it fits: The entire pipeline (research -> discussion -> decision) maps
      naturally to a directed graph. LangGraph's built-in state management
      would solve the memory problem.
    * Consideration: Part of the LangChain ecosystem; strong community support.

  SWARM (OpenAI, experimental)
    * Lightweight agent handoff framework
    * Agents can transfer control and context to each other
    * Minimal overhead, function-calling based
    * Why it fits: Simple upgrade path for the current architecture. Could be
      used to let the decider "call back" to a research agent for clarification.
    * Consideration: Still experimental; OpenAI-specific.

  CAMEL (camel-ai.org)
    * Role-playing framework for autonomous cooperation between agents
    * Supports structured debates and negotiation protocols
    * Why it fits: Directly applicable to the discussion phase — agents could
      negotiate and find consensus rather than just broadcasting opinions.

--------------------------------------------------------------------------------
8.2 LLM Orchestration and Workflow Engines
--------------------------------------------------------------------------------

  LANGCHAIN (langchain.com)
    * Comprehensive framework for LLM application development
    * Chains, agents, tools, memory, and retrieval (RAG) components
    * Supports 100+ LLM providers
    * Why it fits: Provides the plumbing for tool use, memory, and multi-model
      support that would need to be built manually otherwise.

  LLAMA-INDEX (llamaindex.ai)
    * Data framework for LLM applications with strong RAG capabilities
    * Advanced indexing, retrieval, and query engines
    * Financial data connectors available
    * Why it fits: Ideal for building the vector memory / RAG system for
      historical report retrieval and market event recall.

  HAYSTACK (deepset.ai)
    * End-to-end framework for building RAG pipelines
    * Modular, production-ready components
    * Strong evaluation and monitoring tools
    * Why it fits: Alternative to LlamaIndex for the RAG component, with
      better built-in evaluation capabilities.

  PREFECT / AIRFLOW (prefect.io / airflow.apache.org)
    * Workflow orchestration platforms for data pipelines
    * Scheduling, monitoring, retry logic, dependency management
    * Why it fits: Replace the current batch/PowerShell scheduling with a
      proper orchestration engine. Provides monitoring, alerting, and
      failure recovery.

  TEMPORAL (temporal.io)
    * Durable execution platform for long-running workflows
    * Automatic retries, timeouts, and state persistence
    * Why it fits: The multi-phase pipeline with API calls that can fail
      is a perfect use case for Temporal's durable execution model.

--------------------------------------------------------------------------------
8.3 Agentic Operating Systems and Platforms
--------------------------------------------------------------------------------

  SEMANTIC KERNEL (Microsoft)
    * AI orchestration SDK with planner and plugin architecture
    * Supports multiple LLM providers
    * Enterprise-grade with Azure integration
    * Why it fits: Provides a structured way to define agent skills (plugins)
      and orchestrate them with an AI planner.

  LETTA (formerly MemGPT, letta.com)
    * Stateful AI agents with persistent memory management
    * Long-term memory, working memory, and archival memory
    * Self-editing memory for agent self-improvement
    * Why it fits: Directly addresses the "no persistent memory" gap. Agents
      could maintain a growing understanding of the user's portfolio, market
      patterns, and advisory outcomes.

  COMPOSIO (composio.dev)
    * Tool integration platform for AI agents

8.4 Technical Methods and Algorithms
--------------------------------------------------------------------------------

  RETRIEVAL-AUGMENTED GENERATION (RAG)
    * Store historical reports, market data, and advisory outcomes in a vector
      database. Retrieve relevant context at query time rather than stuffing
      the last 3 reports into every prompt.
    * Implementation: ChromaDB (local), Pinecone (cloud), or Weaviate (hybrid)
    * Impact: More relevant context, reduced token usage, better memory.

  CHAIN-OF-THOUGHT (CoT) PROMPTING WITH VERIFICATION
    * Enhance agent prompts with explicit reasoning chains and self-verification
      steps. The current self-iteration is a form of this, but could be more
      structured with intermediate verification checkpoints.
    * Impact: Higher quality reasoning, fewer hallucinations.

  MIXTURE OF AGENTS (MoA)
    * Use multiple LLMs in parallel for the same task, then aggregate their
      outputs. A "meta-agent" synthesizes the best response from the candidates.
    * Implementation: Run GPT-4o, Claude, and Gemini in parallel for the
      decider phase; use a final aggregator to select/merge the best output.
    * Impact: Reduced single-model bias, higher quality decisions.

  CONSTITUTIONAL AI / GUARDRAILS
    * Define explicit rules (financial compliance, Shariah constraints) as
      machine-verifiable checks that run after each agent output.
    * Implementation: NeMo Guardrails (NVIDIA) or custom rule engine
    * Impact: Guaranteed constraint compliance, reduced hallucination risk.

  STRUCTURED OUTPUT / FUNCTION CALLING
    * Instead of free-text generation + parsing, use the Gemini API's
      structured output mode (response_schema) to get machine-readable JSON
      directly from the decider agent.
    * Impact: Eliminates the advisory executor's fragile LLM parsing step.

  ENSEMBLE FORECASTING
    * Combine multiple forecasting signals (technical analysis, sentiment
      scores, fundamental ratios) into a weighted ensemble prediction.
    * Implementation: Lightweight gradient boosting (XGBoost/LightGBM) on
      features extracted by research agents.
    * Impact: More robust market predictions.

  CONFIDENCE CALIBRATION
    * Track the historical accuracy of confidence labels (HIGH/MEDIUM) and
      calibrate them over time. If "HIGH confidence" recommendations are only
      correct 60% of the time, the system should learn to adjust.
    * Impact: More trustworthy confidence signals.

8.8 Learning and Self-Improvement Methods
--------------------------------------------------------------------------------

As outlined in the project's foundational document, the learning roadmap has
five progressive stages. Here are the concrete tools and methods for each:

  STAGE 1: STATIC ADVISOR WITH FULL DECISION LOGGING (CURRENT STATE)
    Status: Implemented. The evaluation_log.csv and advisory_actions_log.json
    capture the necessary data.

  STAGE 2: PROMPT-LEVEL ADAPTIVE FEEDBACK
    Method: Compute performance deltas between advisory and current portfolios.
    Summarize as structured text and inject into agent prompts.
    Tools: pandas for computation, custom template injection.
    Timeline: Achievable in 1-2 weeks of development.

  STAGE 3: OFFLINE EVALUATION AND REWARD MODELING
    Method: Define a reward function combining risk-adjusted return, constraint
    compliance, and consistency. Score historical advisory actions offline.
    Tools: QuantStats for metrics, custom reward function, matplotlib for
    reward distribution visualization.
    Timeline: 2-4 weeks.

  STAGE 4: CONTROLLED RLVR EXPERIMENTS
    Method: Frame as contextual bandit or sequential decision process. Use
    verbal/scalar rewards to adjust agent behavior.
    Tools: Weights & Biases for experiment tracking, custom RL loop.
    Key Challenge: Designing a reward function that doesn't optimize for
    short-term performance at the expense of long-term stability.
    Timeline: 1-3 months of research and experimentation.

  STAGE 5: OPTIONAL MARL-BASED OPTIMIZATION
    Method: Assign partial credit to each agent in the pipeline. Use
    cooperative multi-agent RL to optimize coordination.
    Tools: PettingZoo (multi-agent RL environments), RLlib (distributed RL).
    Key Challenge: Credit assignment — determining which agent's input led
    to which outcome.
    Timeline: Research project (6+ months).
